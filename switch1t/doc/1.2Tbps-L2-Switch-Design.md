## 1. 系统总体架构

### 1.1 关键设计参数

| 参数 | 规格 |
|------|------|
| 总带宽 | 1.2Tbps (48×25Gbps 全双工) |
| 端口数 | 48×25GE SFP28 |
| MAC表容量 | 32K 条目 |
| 转发模式 | Store-and-Forward / Cut-Through 可配置 |
| 最小包处理能力 | 1785.7 Mpps (64B包，线速) |
| 单端口最大吞吐 | 37.2 Mpps (64B包) |
| 企业特性 | VLAN / QoS / ACL / 流控 |

### 1.2 核心架构选型：共享内存交换矩阵 (Shared Memory Switch Fabric)

```
┌────────────────────────────────────────────────────────────────────┐
│                        Switch Core (1.2Tbps)                       │
│  ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐        │
│  │ Ingress  │──▶│  Lookup  │──▶│  Memory  │──▶│  Egress  │        │
│  │ Pipeline │   │  Engine  │   │  Manager │   │ Scheduler│        │
│  └──────────┘   └──────────┘   └──────────┘   └──────────┘        │
│       ▲              ▲              ▲              │               │
│       │              │              │              ▼               │
│  ┌────┴────┐    ┌────┴────┐   ┌────┴────┐   ┌──────────┐         │
│  │  Port   │    │  MAC    │   │ Packet  │   │   QoS    │         │
│  │ Arbiter │    │  Table  │   │ Buffer  │   │ Queues   │         │
│  └─────────┘    └─────────┘   └─────────┘   └──────────┘         │
└────────────────────────────────────────────────────────────────────┘
```

**设计理由**：
- 共享内存架构在中等端口密度(48口)下效率最优
- 避免 Crossbar 的 N² 复杂度
- 支持组播/广播的天然优势（指针复制而非数据复制）

## 3. MAC 查表引擎设计

### 3.1 表结构设计

**容量需求**: 32K 条目

**存储方案**: Hash + SRAM 混合

```
MAC表条目格式 (72bit):
┌──────────────────────────────────────────────────────────┐
│ MAC Addr │ VID  │ Port │ Static │ Age │ Valid │ Reserved │
│  48bit   │ 12bit│ 6bit │  1bit  │ 2bit│  1bit │   2bit   │
└──────────────────────────────────────────────────────────┘

存储组织:
- 4-way Set-Associative
- 8K Sets × 4 Ways = 32K 条目
- Set Index = CRC16(MAC[47:0]) XOR VID[11:0]
```

### 3.2 查表流水线

```
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│ Hash    │──▶│ SRAM    │──▶│ Compare │──▶│ Result  │
│ Compute │   │ Read    │   │ Match   │   │ Output  │
│ 1 cycle │   │ 2 cycle │   │ 1 cycle │   │ 1 cycle │
└─────────┘   └─────────┘   └─────────┘   └─────────┘
                                 │
                                 ▼
                          ┌─────────────┐
                          │ Miss: Flood │
                          │ to VLAN     │
                          └─────────────┘

查表吞吐: 500M次/秒 (2倍于线速需求)
```

### 3.3 MAC 学习机制

```
学习触发: SMAC 在表中未命中
学习队列: 深度512，防止学习风暴
学习限制: 每端口每秒最大1K次学习 (防MAC洪泛攻击)

老化机制:
- 软件定时扫描 (默认300秒)
- 硬件辅助: 2bit Age计数器
- 访问时重置Age，扫描时递减
```

## 4. 内存管理模块设计

### 4.1 共享缓冲池架构

**关键挑战**: 1.2Tbps 带宽下的内存访问瓶颈

**设计决策**: 采用纯片内SRAM方案，不使用外置DDR

```
DDR方案分析 (已否决):
┌─────────────────────────────────────────────────────────────┐
│ 问题: DDR访问延迟 50-80ns，约为SRAM的10-20倍               │
│ 影响: Cut-Through模式目标 <500ns，DDR会成为瓶颈            │
│ 结论: 对于数据中心L2交换机，DDR延迟不可接受                │
└─────────────────────────────────────────────────────────────┘

缓冲容量需求分析:
- 典型数据中心突发: 10-100μs
- 4:1拥塞，100μs突发: 4 × 25Gbps × 100μs = 1.25MB
- BDP经验公式: RTT(250μs) × 25Gbps × 48 / sqrt(48) ≈ 5.4MB
- 设计容量: 8MB (覆盖典型场景，可扩展至16MB)

缓冲池规格 (纯片内SRAM):
┌─────────────────────────────────────────────────────────────┐
│              Unified Packet Buffer (片内SRAM)               │
│                                                             │
│    容量: 8MB (可扩展至16MB)                                 │
│    组织: 64K Cells × 128B/Cell                              │
│    Bank: 16 Banks 并行访问                                  │
│    延迟: < 5ns (确定性，无抖动)                             │
│                                                             │
│    突发吸收能力:                                            │
│    - 线速缓冲: 8MB / 1.2Tbps = 53μs                        │
│    - 4:1拥塞: 53μs × 4 = 212μs 突发吸收                    │
└─────────────────────────────────────────────────────────────┘

纯SRAM方案优势:
- 延迟确定性: 所有报文延迟一致，无DDR访问抖动
- 架构简化: 无需冷热数据迁移逻辑
- 功耗降低: 省去DDR PHY功耗 (约3-5W)
- Cut-Through友好: 真正实现 <500ns 转发延迟
```

### 4.2 Cell 分配器

```
空闲Cell管理:
- 采用链表 + Bitmap 混合结构
- 4个并行分配引擎 (支持4个同时分配请求)

分配流程:
1. 从空闲链表头取Cell ID
2. 更新Bitmap标记为已占用
3. 返回Cell ID给Ingress
4. 吞吐: 2G cells/s (4路并行 × 500MHz)

释放流程:
1. Egress发送完成后返回Cell ID
2. 清除Bitmap
3. 加入空闲链表尾
```

### 4.3 链表数据结构设计

#### 4.3.1 Cell元数据结构

```
每个Cell需要维护的元数据 (32bit):
┌────────────────────────────────────────────────────────────┐
│  Next_Ptr  │  Ref_Cnt  │  EOP  │  Valid  │    Reserved    │
│   16bit    │   3bit    │  1bit │  1bit   │     11bit      │
└────────────────────────────────────────────────────────────┘

字段说明:
- Next_Ptr (16bit): 指向下一个Cell，支持64K Cells寻址
- Ref_Cnt (3bit): 引用计数，支持最多7个组播副本
- EOP (1bit): End of Packet标记，表示报文最后一个Cell
- Valid (1bit): Cell有效标记
- Reserved (11bit): 预留扩展

元数据存储:
- 独立SRAM存储，与报文数据分离
- 64K entries × 32bit = 256KB
- Dual-port设计，支持读写并行
```

### 4.4 报文存储策略

```
Store-and-Forward 模式:
┌────────┬────────┬────────┬────────┐
│ Cell 0 │ Cell 1 │ Cell 2 │ Cell 3 │ ─▶ 完整报文
└────────┴────────┴────────┴────────┘
      │
      ▼ 整包接收完成后入队列

Cut-Through 模式:
┌────────┐
│ Cell 0 │ ─▶ 收到64B (含目的MAC) 后立即查表转发
└────────┘
      │ 后续Cell流式写入，边写边读
      ▼
┌────────┬────────┬────────┐
│ Cell 1 │ Cell 2 │ Cell 3 │
└────────┴────────┴────────┘

模式切换:
- 可按端口配置
- CRC错误报文回退到S&F丢弃
```

### 4.5 内存访问调度

```
多Bank并行访问:
- SRAM分16 Banks，避免访问冲突
- Cell ID[3:0] 决定Bank选择
- 支持16个并行读/写操作

读写优先级:
1. Egress读取 (最高，保证不欠速)
2. Ingress写入 (次高)
3. 组播指针复制 (最低)

带宽计算:
- 16 Banks × 512bit × 500MHz = 4Tbps 内存带宽
- 裕量: 4Tbps / 1.2Tbps ≈ 3.3x
```

## 5. Egress 调度与输出模块

### 5.1 队列结构

```
每端口8个优先级队列 (符合802.1p):
Port 0: Q7(最高) ─ Q6 ─ Q5 ─ Q4 ─ Q3 ─ Q2 ─ Q1 ─ Q0(最低)
Port 1: Q7 ─ Q6 ─ Q5 ─ Q4 ─ Q3 ─ Q2 ─ Q1 ─ Q0
  ...
Port 47: Q7 ─ Q6 ─ Q5 ─ Q4 ─ Q3 ─ Q2 ─ Q1 ─ Q0

队列实现:
- 总队列数: 48端口 × 8队列 = 384 队列
- 队列深度: 基于Cell指针链表，动态共享
- 队列描述符: {Head_Ptr, Tail_Ptr, Length, Drop_Cnt}
```

### 5.2 调度算法

```
两级调度架构:

Level 1 - 端口内优先级调度:
┌─────────────────────────────────────────┐
│  Strict Priority + WRR 混合调度         │
│  - Q7/Q6: Strict Priority (控制/实时)   │
│  - Q5~Q0: WRR 权重调度                  │
│    权重比例: 8:4:2:2:1:1               │
└─────────────────────────────────────────┘

Level 2 - 跨端口带宽公平:
┌─────────────────────────────────────────┐
│  Deficit Weighted Round Robin (DWRR)   │
│  - 每端口 Quantum = MTU × Weight        │
│  - Deficit记录剩余额度                   │
│  - 保证长期公平性                        │
└─────────────────────────────────────────┘
```

### 5.3 拥塞控制

```
主动队列管理 (AQM):

尾部丢弃 (Tail Drop):
- 队列长度 > Max_Threshold 时丢弃新包

加权随机早期检测 (WRED):
- 队列长度在 Min_Th ~ Max_Th 之间时概率丢弃
- 丢弃概率 = (Len - Min_Th) / (Max_Th - Min_Th) × Max_Prob
- 不同颜色(Green/Yellow/Red)差异化门限

流量整形 (Shaping):
- 每端口支持出向速率限制
- 令牌桶算法: CIR/CBS + PIR/PBS
- 粒度: 1Mbps
```

### 5.4 Egress Pipeline

```
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│ Dequeue │──▶│ Memory  │──▶│ Egress  │──▶│ TX MAC  │
│ Decision│   │ Read    │   │ Modify  │   │ Output  │
└─────────┘   └─────────┘   └─────────┘   └─────────┘
                                │
                                ▼
                         ┌──────────────┐
                         │ VLAN Tag操作  │
                         │ Push/Pop/Swap│
                         └──────────────┘

Egress修改动作:
- VLAN Tag 添加/删除/替换
- PCP 重标记
- 统计计数器更新
```

## 6. 特殊处理机制

### 6.1 组播/广播处理

```
组播复制:
- 不复制报文数据，仅复制Cell指针
- 使用引用计数: 每Cell维护Ref_Count
- 最后一个Egress读取后释放Cell

广播处理:
- 泛洪到VLAN内所有端口 (排除源端口)
- 使用VLAN成员位图加速判断

组播组表:
- 256个组播组
- 每组48bit端口成员位图
```

### 6.2 环路防护

```
源端口过滤:
- 防止报文回送到源端口

BPDU处理:
- STP BPDU上送CPU
- 支持端口STP状态: Disabled/Blocking/Learning/Forwarding

端口状态机:
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ Disabled │──▶│ Blocking │──▶│ Learning │──▶│Forwarding│
└──────────┘   └──────────┘   └──────────┘   └──────────┘
    │               │               │               │
    └───────────────┴───────────────┴───────────────┘
                    (状态可回退)
```
